The GoMoney API uses two authentication layers that work together to guarantee the identity and authorization of whoever operates the integration: an **authenticated session** through a JWT token and an **integration key** encoded in Base64. Both must be present and belong to the same integrator user for requests to be accepted.

## Authentication Model

GoMoney API security is based on two complementary factors:

| Layer | Mechanism | Purpose |
|---|---|---|
| **Session Authentication** | `Authorization: Bearer <JWT>` | Confirms who is operating the active session |
| **Integration Authentication** | `secret-key: <Base64>` | Proves that the call comes from a specific authorized integrator |

The two layers work together: the session identifies the logged-in user, while the integration key ensures that the request does not rely exclusively on the session and is linked to the correct integrator. Neither one alone is enough to access protected routes.

<Note>
  The Bearer token and the `secret-key` must belong to the **same integration user**. Using credentials from different users will result in an authentication error.
</Note>

---

## Integration Access Requirements

To use the API, the user must meet the following access requirements:

- **Access type:** `INTEGRATION` — the user profile must be an integration profile, not an end-user profile.
- **Active permission:** `INTEGRATION_ONLY` — the specific integration permission must be enabled on the account.
- **Dedicated credentials:** the integrator user must have their own email and password, used exclusively for API login.

<Warning>
  Never share the integrator user's email and password with third parties. These credentials grant operational access to the API and must be treated with the same level of secrecy as the integration keys.
</Warning>

## Session Authentication

The operational session is obtained at login time and must be reused in protected customer and balance addition routes. To start a session, send a `POST` request to the login endpoint:

```http
POST https://gmc-api.gomoney.me/api/users/integration/login
Content-Type: application/json

{
  "email": "{{userEmail}}",
  "password": "{{userPassword}}"
}
```

On success, the API returns an object containing the JWT token and session information:

```json
{
  "token": "integration-jwt-token",
  "title": "Successfull login.",
  "expiresIn": 1783191600000,
  "userId": "66f0b1b5c1f7b7dc0ce20001"
}
```

| Field | Type | Description |
|---|---|---|
| `token` | `string` | JWT token to use in the `Authorization: Bearer` header |
| `title` | `string` | Login confirmation message |
| `expiresIn` | `number` | Unix timestamp (ms) indicating token expiration |
| `userId` | `string` | Unique identifier of the integrator user on the platform |

<Tip>
  Store the `expiresIn` value and implement automatic token renewal logic before it expires to avoid interruptions in production operations.
</Tip>

## Building the `secret-key`

The integration key is generated by combining the `clientKey` and `clientSecret` provided by GoMoney support, separated by a colon (`:`) and encoded in Base64:

```js
// JavaScript / Node.js
const secretKey = btoa(`${clientKey}:${clientSecret}`);
```

```python
# Python
import base64
secret_key = base64.b64encode(f"{client_key}:{client_secret}".encode()).decode()
```

```bash
# cURL / Shell
SECRET_KEY=$(echo -n "${CLIENT_KEY}:${CLIENT_SECRET}" | base64)
```

<Note>
  The `clientKey` and `clientSecret` are delivered by the GoMoney support team by email after the integrator user is created and approved. See the [Requesting Integration Keys](#requesting-integration-keys) section for more details.
</Note>

## Sending Credentials in Headers

Every request to protected routes must include both authentication headers at the same time:

```http
Authorization: Bearer {{accessToken}}
secret-key: {{secretKeyBase64}}
```

**Complete authenticated request example:**

```http
GET https://gmc-api.gomoney.me/api/customers
Authorization: Bearer eyJhbGciOiJIUzI1NiIsInR5cCI6IkpXVCJ9...
secret-key: Y2xpZW50S2V5OmNsaWVudFNlY3JldA==
Content-Type: application/json
Origin: https://seu-dominio-aprovado.com.br
```

<Warning>
  The `Origin` header must exactly match the domain registered and approved by GoMoney support. Requests originating from unauthorized domains will be rejected.
</Warning>

## Validation Rules

The GoMoney API performs the following checks before processing any request to protected routes:

- The `Authorization` header must be present and contain a valid, non-expired Bearer JWT token.
- The `secret-key` header must be present and contain the correct Base64 value for `clientKey:clientSecret`.
- The Bearer token and the `secret-key` must belong to the **same integration user**; mixed combinations are rejected.
- The session user must have the `INTEGRATION` access type with the `INTEGRATION_ONLY` permission active.
- The request `Origin` header must match the domain previously approved by GoMoney support.
- The JWT token must not be expired. Expired tokens return a `401 Unauthorized` error.

## Requesting Integration Keys

To obtain the `clientKey` and `clientSecret`, contact GoMoney support and provide the following information:

1. **Integrator user:** email and details of the integration user created on the platform.
2. **API access:** confirmation that the `INTEGRATION` access type was created and configured.
3. **Environment:** specify whether the request is for the **sandbox** or **production** environment.
4. **Origin/Domain:** provide the domain that will be used in the `Origin` request header.
5. **Receiving wallet:** indicate the wallet that will receive integration operations.
6. **Registration confirmation:** confirm that the wallet is properly registered and approved on the platform.

Support returns the `clientKey` and `clientSecret` by email after validating the information.

## Secure Storage Best Practices

<Warning>
  **Never publish your credentials in code repositories**, even private ones. Leaks in `git` history are one of the main causes of API key compromise.
</Warning>

Follow the guidelines below to protect your integration credentials:

- **Environment variables:** store `clientKey`, `clientSecret`, email, and password in environment variables (`.env`) and never commit those files.
- **Vault / secret manager:** in production environments, use solutions such as HashiCorp Vault, AWS Secrets Manager, or equivalents.
- **Do not expose in logs:** avoid recording key values, tokens, or the `secret-key` in application logs or monitoring systems.
- **Rotate on exposure:** if any credential is accidentally exposed, contact GoMoney support immediately to request key rotation.
- **Least access:** restrict access to credentials only to the parts of the system that actually need them.

<Tip>
  Consider adopting a code review process that includes secret scanning before any repository publication to prevent credentials from being exposed accidentally.
</Tip>
