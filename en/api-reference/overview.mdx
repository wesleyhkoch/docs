This section documents the **GoMoney integration API**, allowing partners to perform programmatic operations such as authentication, customer management, and balance addition. All requests must be sent to the base URL below.

<Note>
  **Base URL**

  ```
  https://gmc-api.gomoney.me/api
  ```
</Note>

<Tip>
  Every endpoint in this reference has an **interactive playground** — fill in the fields with your own data and click **Try it** to send a real request straight from your browser. No Postman, Insomnia, or Swagger required.
</Tip>

## Access Prerequisites

Before consuming any API route, make sure the following requirements are met:

- **INTEGRATION access type** created on the GoMoney platform.
- **`INTEGRATION_ONLY` permission** active on the integration access.
- **Email and password** for the integration access to authenticate via API.
- **`clientKey` and `clientSecret`** provided by GoMoney support.
- **Approved domain** to be used in the `Origin` request header.
- **Registered and approved wallet** for the GMC receiving flow.

## Recommended Technical Sequence

Follow the order below to ensure a correct, unblocked integration:

1. **Create the integration access** on the GoMoney platform with the `INTEGRATION` type.
2. **Confirm that the `INTEGRATION_ONLY` permission** is active on the created access.
3. **Request from support** the `clientKey`, `clientSecret`, and approval for the origin domain.
4. **Confirm that the operational wallet** is registered and approved for receiving.
5. **Log in through the API** using the integration access credentials.
6. **Build the `secret-key`** by Base64-encoding the `clientKey:clientSecret` pair: `btoa(clientKey:clientSecret)`.
7. **Register or locate the customer** that will serve as the reference in financial reconciliation.
8. **Create a balance addition** by providing the corresponding customer's `customerIdentifier`.
9. **Query the list or detail** of the balance addition to monitor the operational lifecycle.

## Usage Notes

<Note>
  The **access token** used in requests must be obtained exclusively by logging in with an `INTEGRATION` access that has the `INTEGRATION_ONLY` permission active.
</Note>

<Note>
  The **authenticated session**, the **`secret-key`**, and the **authorized domain** must represent the same integration. Mixing credentials from different origins will result in authorization errors.
</Note>

<Note>
  The **operational wallet** must be registered and approved before any balance addition operation can succeed.
</Note>

<Note>
  The **`customerIdentifier`** field is required in all balance addition operations and must match a customer previously registered in the integration.
</Note>
